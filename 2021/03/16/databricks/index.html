What is Delta Lake

Delta Lake is a robust storage solution designed specifically to work with Apache Spark.

A New Data Management Paradigm - Lakehouse

Data management solutions have evolved over time:

  • Data Warehouses: great in handling large-size structured data, but not the unstructured or semi-structured.
  • Data Lakes: repositories for raw data in a variety of formats. Good for storing, not good for analyzing
  • Lakehouses: implement data warehouse data structures and data management features directly on the kind of low-cost storage used for data lakes.

Delta Lake and Apache Spark

What is Delta Lake

Delta Lake is a technology for building robust data lakes and is a component of building your cloud data platform, and specifically designed to work with Apache Spark.

Delta Lake’s Architecture

  • Delta table: atomic data stored in data lake (azure data lake, Amazon S3 for example), and a table registered in Metastore lives in Databricks workspace to include the locations of the delta files, schema, and properties of the table.
  • Delta optimization engine: use optimized Apache Spark to read and write to Delta Lake.
  • Delta Lake storage layer: keep all of your data in files in object storage
  • Delta architecture design pattern: landing data in successively cleaner delta lake tables

Why Delta Lake

  • Separation of computing and storage
  • Support streaming, replacing the lambda architecture pattern
  • ACID (Atomicity, Consistency, Isolation, Durability), fast

A Single Source of Truth for Business (SSOT)

SSOT: the single source of data that is used by everyone in an organization when mining data to make business decisions.
EDSS: An enterprise decision support system, or EDSS, is an information system used by an organization to help decision-making. An EDSS contains all available transactions performed during the history of a business, as well as all versions of information needed to perform those transactions. An EDSS is most effective if there is an SSOT for data and can perform Online Analytical Processing (OLAP). Good for high throughput.
ODS: An Operational Data Store only contains the latest version of a particular data point. An ODS can perform the online transaction processing. Good for high concurrency.

Complete data system, an ETL process pulls data from the ODS to be loaded into the EDSS.

Levels of Data in an Architecture Data System

  1. ODS
  • day-to-day data that an organization collects. The needs of analysts who may be looking at this data are secondary. For example, a credit score record.
  • MySQL can be associated with this layer since it is often deployed as the primary database for a web application.
  • Kafka is a stream processing software and can also be associated with an operational layer
  1. Atomic (EDSS)
  • This is known as the storage layer. This is the single source of truth. For example, all the transaction credit score record overtime
  • A data warehouse (redshift, snowflake) is most traditionally associated with the atomic layer, but it can also be a silver table with object storage
  1. Departmental (data mart)
  • Designed to make specific needs, also known as the query layer, is a subset of atomic data. For example, new customers by month
  • Gold table with object storage
  1. Individual
  • Temporary, often used to answer highly-specific questions, generated by SQL or spark command.

Enterprise Decision Support System Architectures

Several ways to implement an EDSS

On-premises

  • Highly optimized, massively parallel processing system, with well-established technology
  • Not elastic, hard to move data from a given location, relies on structured data, and expensive

Cloud-based Data Warehouse

  • This architecture often includes a low-cost raw layer as part of the ETL process.
  • Separation of computing and storage, elastic resource allocation; optimized for query throughput
  • Not easy to move data, structured data only; data lake not queryable

Lakehouse

  • Low data gravity because SSOT is data in the data lake
    no limitation on data structure
  • Not designed for high concurrency (however it is for high data throughput), so it handles large data well, but not many users
